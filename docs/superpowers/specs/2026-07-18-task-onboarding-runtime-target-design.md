# 新任务接入与模型调用通道设计

## 1. 目标

AI Task Center 的新建任务不再要求用户从固定处理器列表中选择任务类型，也不再把编码工具作为任务配置。新任务先保存业务信息和数据来源，再在任务接入阶段选择唯一的模型调用通道，并通过通用提示词让用户在任意外部编码工具中实现任务专属处理器。

本设计只保证新任务流程。旧任务、旧载荷和旧字段的运行兼容不属于验收范围。

## 2. 已确认原则

- Codex CLI、Antigravity CLI、MiMo API 和其他模型 API 都是正式任务运行时的模型调用通道。
- 模型调用通道只负责在任务过程中返回文本、评分 JSON、音频或其他模型输出，不负责生成或修改 AI Task Center 代码。
- 任务中心只生成通用接入提示词。用户可以把提示词复制到 Codex、Antigravity 或其他外部编码窗口，平台不记录使用了哪个编码工具。
- 每个任务只允许选择一个模型调用通道。
- 新建任务时不选择任务处理器，也不选择模型调用通道。
- 任务处理器由后续接入代码实现并注册。
- 旧 `cli_id`、`onboarding_cli_id` 数据库列保留并放宽为可空，但新代码不再映射或写入其业务值，新记录对应列为 `null`。
- 不为旧任务增加来源表、载荷或旧 CLI 字段的推断回退。

## 3. 新建任务

### 3.1 页面字段

新建和编辑基础信息页面只保留：

- 任务名称。
- 所属项目。
- 关联数据库。
- 来源表。
- 任务描述。

页面删除：

- 任务处理器。
- 运行调用通道。
- 接入代码 CLI。

### 3.2 初始状态

新任务保存后：

```text
handler_key = null
executor_type = null
executor_id = null
onboarding_step = TARGET_SELECTION
onboarding_status = ACTIVE
```

任务基础信息保存不校验模型调用通道，也不校验处理器。

## 4. 任务接入流程

任务接入流程调整为八个节点：

1. 选择模型调用通道。
2. 定制结果代码。
3. 验证任务结果。
4. 生成全部结果。
5. 定制批次代码。
6. 验证任务批次。
7. 生成全部批次。
8. 任务列表可用。

对应步骤枚举：

```text
TARGET_SELECTION
RESULT_CODE
RESULT_VALIDATION
RESULT_GENERATION
BATCH_CODE
BATCH_VALIDATION
BATCH_GENERATION
READY
```

### 4.1 选择模型调用通道

任务接入抽屉从安全调用通道目录加载可选目标。用户必须选择一个已启用目标，目录只返回：

- `type`：`CLI` 或 `AI_PROVIDER`。
- `id`。
- 显示名称。
- 协议。
- 能力列表。
- 启用状态。

目录不得返回 API Key、环境变量、完整命令参数或其他敏感配置。

选择成功后保存 `executor_type` 和 `executor_id`，并将接入步骤推进到 `RESULT_CODE`。

### 4.2 接入提示词

结果代码和批次代码提示词包含：

- 任务 ID、名称和描述。
- 数据库及来源表信息。
- 所选模型调用通道的 ID、类型、协议和非敏感能力。
- 平台分配的稳定处理器 Key：`task_config_<taskConfigId>`。
- 当前阶段所需的处理器接口。
- Java、React、Python Worker 的职责边界。
- 测试、禁止真实模型调用和回填命令要求。

提示词不绑定 Codex 或 Antigravity，也不包含任何模型密钥。

## 5. Python Worker 处理器注册表

### 5.1 处理器描述

Python Worker 新增统一处理器注册表。每个处理器提供安全描述信息：

```text
handlerKey
requiredCapability
supportsResultGeneration
supportsSingleValidation
supportsBatchBuild
supportsBatchExecution
```

处理器实现可以提供：

- 生成任务结果。
- 构建批次载荷。
- 执行单条验证。
- 执行正式批次。
- 解析模型返回内容。
- 回填来源业务表。

处理器通过标准模型执行接口使用任务快照中的 `executor_type` 和 `executor_id`，不能自行选择其他模型通道，也不能自动跨通道回退。

### 5.2 注册与发现

新任务处理器使用平台预分配的 `task_config_<id>` 作为 Key。外部编码工具实现代码、注册处理器、补充测试并重启 Python Worker，然后使用现有 `task-workflow report` 回填 `CODE_READY`。

回填命令不新增处理器参数，避免让外部工具自行声明或替换任务处理器 Key。

Python Worker 提供只读安全处理器目录接口，Java 可以按 Key 查询：

- 是否已经注册。
- 当前阶段需要的函数是否可用。
- 处理器声明的模型能力。

处理器目录不得返回函数源码、密钥、连接密码或模型私有配置。

### 5.3 现有处理器

现有评分和 TTS 实现迁入同一注册表，作为注册机制的内置实现和测试样例。新任务不从这两个处理器中选择，也不根据来源表自动使用它们。

## 6. 代码就绪回填

Java 收到结果阶段 `CODE_READY` 时执行：

1. 验证当前步骤、一次性 Token 和阶段。
2. 计算预期处理器 Key `task_config_<id>`。
3. 查询 Python Worker 安全处理器目录。
4. 确认结果生成和单条验证接口已注册。
5. 确认处理器 `requiredCapability` 包含在所选模型调用通道能力中。
6. 将预期 Key 写入任务配置 `handler_key`。
7. 推进到 `RESULT_VALIDATION`。

Java 收到批次阶段 `CODE_READY` 时额外确认：

- 批次构建接口已注册。
- 批次执行接口已注册。

任一检查失败时不推进步骤，只保存可展示的错误摘要。

## 7. 正式执行

正式结果和批次创建时必须保存以下快照：

```text
handler_key
executor_type
executor_id
```

Python Worker 严格按 `handler_key` 从注册表选择处理器，再按 `executor_type + executor_id` 加载唯一模型调用通道。

以下情况立即失败，不做历史推断：

- `handler_key` 为空。
- 处理器未注册。
- `executor_type` 或 `executor_id` 为空。
- 模型调用通道不存在或已禁用。
- 模型能力与处理器要求不匹配。
- 当前执行阶段所需处理器函数未实现。

失败重试继续使用相同快照，不自动更换模型调用通道。

## 8. 更换模型调用通道

- 在结果代码尚未完成前，可以重新选择模型调用通道，并生成新的提示词和一次性 Token。
- 结果代码完成后更换通道，会将任务接入流程重置到 `RESULT_CODE`，清除当前及下游代码就绪状态。
- 新通道必须启用。
- 后续代码回填时重新检查处理器能力与新通道能力。

## 9. 字段与兼容策略

继续使用并允许为空：

```text
handler_key
executor_type
executor_id
```

以下历史列保留但改为可空，新领域模型和新接口不再映射、读取或写入：

```text
tb_task_config.cli_id
tb_task_config.onboarding_cli_id
tb_task_result.cli_id
tb_task_run.cli_id
tb_task_execution_log.cli_id
```

启动时执行幂等的非破坏性约束调整，只移除这些历史列的 `NOT NULL` 约束，不删除列或旧数据。新记录对应列保持 `null`。

新代码删除以下行为：

- 根据来源表推断评分或 TTS。
- 根据 Payload `taskType` 推断处理器。
- 根据旧 `cli_id` 推断模型调用通道。
- 新建或编辑任务时选择固定处理器。
- 保存、展示或验证接入代码 CLI。

旧任务是否能够继续执行不作为本次设计目标，也不新增数据迁移或批量修复。

## 10. 错误处理

- 未选择模型调用通道时，不允许生成接入提示词。
- 调用通道不存在或已禁用时，不推进 `TARGET_SELECTION`。
- Python Worker 不可用时，代码就绪回填失败并保留当前步骤。
- 处理器未注册或阶段函数不完整时，返回明确缺失项。
- 能力不匹配时同时返回处理器所需能力和所选通道能力。
- 运行时注册或目标缺失时，任务结果或批次标记失败并保存明确原因。
- 错误响应和日志不得包含 API Key、连接密码或完整敏感配置。

## 11. 测试范围

### 11.1 Java

- 新建任务不需要处理器、模型通道或 CLI。
- 新任务默认进入 `TARGET_SELECTION`。
- 只能选择存在且已启用的模型调用通道。
- 未选择通道不能生成结果代码提示词。
- 提示词包含安全目标元数据和预期处理器 Key，不包含密钥。
- `CODE_READY` 回填验证 Worker 注册信息和阶段函数。
- 能力不匹配时不推进接入步骤。
- 新代码不使用旧 `cli_id` 作为执行目标回退。

### 11.2 Python Worker

- 内置评分和 TTS 处理器通过统一注册表发现。
- 可以注册 `task_config_<id>` 处理器。
- 结果生成、单条验证和批次执行统一从注册表路由。
- 未注册处理器明确失败。
- 缺少模型调用通道明确失败。
- 不从来源表或 Payload 推断处理器。
- 安全处理器目录不暴露敏感信息。

### 11.3 React

- 新建和编辑页面不显示三个旧字段。
- 任务接入抽屉展示八节点流程。
- 第一个节点可以选择一个模型调用通道并展示安全元数据。
- 未选择目标时后续节点保持锁定。
- 接入完成后只读展示处理器、模型调用通道和所需能力。
- 页面不再提交 `cliId` 或 `onboardingCliId`。

### 11.4 集成验收

使用模拟模型响应完成以下闭环，不调用真实 CLI、AI 或 TTS：

```text
创建新任务
→ 不选择处理器或模型
→ 进入任务接入
→ 选择一个模型调用通道
→ 复制通用提示词
→ 注册 task_config_<id> 处理器
→ CODE_READY 回填通过
→ 完成结果与批次验证
→ 正式任务严格使用所选模型调用通道
```

## 12. 非目标

- 不兼容或修复旧任务运行数据。
- 不物理删除旧数据库列。
- 不让任务中心自动启动编码 CLI。
- 不记录用户使用的外部编码工具。
- 不允许一个任务同时选择多个模型调用通道。
- 不在模型调用失败后自动切换其他目标。
