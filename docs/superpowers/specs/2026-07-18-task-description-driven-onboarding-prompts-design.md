# 任务描述驱动的接入提示词设计

## 目标

让任务描述成为“结果代码准备”和“批次代码准备”提示词的核心业务需求，使外部编码模型明确知道要开发什么代码，同时完整保留现有任务接入阶段、仓库边界、安全约束、处理器注册、人工验证和回填命令。

## 表单与数据约束

- 新建和编辑任务时，任务描述必填。
- 去除首尾空白后，任务描述长度必须为 1–2000 个字符。
- React 文本框设置 `required`、`maxLength=2000`、字数统计和清晰的校验提示。
- Java 创建和更新接口执行同样的规范化与长度校验，防止绕过前端。
- 进入结果代码准备或批次代码准备时再次校验。已有任务描述为空时，不生成提示词，并提示用户先编辑任务描述。
- 数据库字段和任务接入状态结构保持不变，不新增历史兼容或自动补写逻辑。

## 提示词结构

两种提示词继续使用当前固定结构：

1. 不可信业务数据前的不可变仓库与架构约束。
2. 明确的业务开发目标。
3. `BEGIN/END UNTRUSTED BUSINESS DATA` 包裹的结构化任务上下文。
4. 不可信业务数据后的重复安全边界。
5. 当前阶段允许执行的代码、测试与服务重启范围。
6. 对应阶段的 `task-workflow report` 命令。

任务描述会同时出现在醒目的“业务开发目标”段落和结构化任务上下文中。安全约束位于任务描述之前和之后，因此任务描述只能作为业务输入，不能覆盖仓库、数据库或模型调用边界。

## 结果代码准备

提示词根据任务描述要求外部编码模型：

- 检查任务选择的来源表结构和真实字段。
- 在 `python-worker` 中为确定性处理器键 `task_config_<taskConfigId>` 实现结果生成回调。
- 根据任务描述完成来源读取、业务筛选、字段映射和结果 JSON 设计。
- 将处理器与所选 `executorType + executorId` 快照写入新结果。
- 添加聚焦测试，但不生成验证数据、不执行任务、不调用 AI/TTS。

## 批次代码准备

提示词使用同一任务描述，并要求外部编码模型：

- 复用结果阶段已经实现的业务载荷和处理器，不重复发明任务含义。
- 实现批次输入构建回调和批次执行回调。
- 根据任务描述定义批次拆分、模型输入、响应解析、逐项状态与错误隔离。
- 所有 CLI、LLM、TTS 或 Provider 调用继续只位于 `python-worker`。
- 添加聚焦测试，但不实际执行任务或调用模型。

## 阶段流转

现有流转保持不变：

`TARGET_SELECTION → RESULT_CODE → RESULT_VALIDATION → RESULT_GENERATION → BATCH_CODE → BATCH_VALIDATION → BATCH_GENERATION → READY`

本改造只影响进入 `RESULT_CODE` 和 `BATCH_CODE` 时生成的提示词内容，以及任务描述的输入校验；不会改变阶段状态、验证标记、处理器就绪校验或回填协议。

## 错误处理

- 空白任务描述：返回明确错误“请先完善任务描述”，不生成提示词。
- 超过 2000 字：前端阻止提交，后端返回相同长度约束错误。
- 提示词 JSON 序列化失败：沿用现有“生成任务引导提示词失败”异常。

## 测试

- `TaskConfigService`：创建和更新拒绝空白、超过 2000 字的任务描述，并保存去除首尾空白后的值。
- `TaskOnboardingPromptBuilder`：结果和批次提示词均突出包含任务描述；阶段专属开发要求不同；现有安全边界、处理器键和回填命令仍存在。
- `TaskOnboardingService`：任务描述缺失时不能进入或读取代码准备提示词。
- React 生产构建：验证必填规则、2000 字限制和字数统计的类型与组件配置有效。

## 非目标

- 不使用大模型自动改写或扩写任务描述。
- 不改变模型调用通道选择方式。
- 不新增任务类型枚举。
- 不执行真实任务、AI 或 TTS。
- 不修改 `task-workflow` 脚本。
